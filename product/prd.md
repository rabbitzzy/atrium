## Problem statement

Bright Horizon Chinese School students attend weekend sessions without personal devices. There is no structured, persistent way to track what each student knows, assign work calibrated to their frontier, or give formative feedback that follows them across visits. Teachers manage this manually, which limits reach and diagnostic depth.

**We are building a shared kiosk system** that gives every student a personalized loop — assess → assign → print → work → submit → reflect → re-plan — in a single 20–40 minute session, with no teacher present at the kiosk.

---

## North star

> A student can walk up to the kiosk, scan their badge, receive a personalized worksheet, work on it, submit it, and see a Debrief — all within one 30-minute session — with no teacher intervention required.

## What we are building

### The flywheel (required for v1)

| Step | Requirement |
|------|-------------|
| 1. Bootstrap eval | Initial placement assigns mastery priors across the seed KC tree |
| 2. Task planning | `nextTask` selects KCs at the student's frontier (not too easy, not too hard) |
| 3. Print Card | Worksheet PDF generated with QR header; costs 1 Leaf; blocked at 0 |
| 4. Student works | Paper; may ask Docent for hints via keyboard (voice deferred to Phase 3) |
| 5. Scan submission | Camera captures completed Card; QR decoded to student + task |
| 6. AI evaluation | Multimodal LLM grades against rubric; structured output |
| 7. Debrief | Shown on screen (digital-first); printable on request (0 Leaves) |
| 8. State update | BKT updates mastery probabilities; +1 Leaf awarded |
| 9. Plan next task | Loop to step 2 |

**Every requirement below is in service of this flywheel. If a feature doesn't make the loop spin faster, smoother, or more diagnostic, it is out of scope.**

## Requirements by feature area

### 1. Student identity and check-in
- Students authenticate via QR badge scan (no password, no PIN in v1)
- Identity and profile are the single source of truth in the BHCS portal — Atrium never holds PII independently
- Session state persists across visits; a student returning after a week resumes from their last skill state

### 2. Skill graph (Blueprint)
- Directed acyclic graph of Knowledge Components (KCs)
- Pilot seed: ~30 KCs across Chinese reading, English reading, and math
- Each KC has: ID, display name (bilingual), parent KC(s), difficulty level, associated rubric
- Cross-subject edges are supported (e.g. a word problem KC tagged to both math and reading)
- Teachers can author new KCs in Phase 2; new KCs require at least one example problem before going live

### 3. Student state
- Per-student, per-KC: mastery probability (0–1), attempt count, last interaction timestamp
- BKT (pyBKT) as the student model; default literature parameters for the pilot
- Radar chart derived from mastery probabilities; readable by student, teacher, and parent
- `nextTask` selects from: KCs near mastery threshold (frontier), with recent-failure KCs weighted higher

### 4. Worksheet (Card)
- Print-ready PDF; A4/Letter; 3–5 problems per page; never mostly blank
- Fixed-template layout: answer regions at fixed coordinates for scan-back alignment (Gradescope pattern)
- QR code in header encodes `student_id + task_id`; scanned on submission
- Bilingual: problem text in Chinese and/or English per KC setting
- Problems generated by LLM constrained to target KC(s), difficulty, and student's interest profile (Phase 2.5+)
- Print gate: requires `leaf_balance >= 1`; returns `402` if blocked

### 5. Leaf economy (print credits)
- Leaf balance per student: floor 0, ceiling 5
- Earn: +1 on Card submission (any quality); +2 on enrollment bootstrap; teacher can grant N with reason
- Spend: -1 on Card print; Debrief print is always free
- Leaf balance visible in kiosk UI at all times (top-right, eco green `#4a7c59`)
- Zero-balance state: Docent message explaining how to earn; never shame language
- All Leaf events logged to `print_events` table with type (earn/spend/grant/refund), amount, reason, session
- Print failure triggers automatic Leaf refund
- Full spec: `/docs/pedagogy/eco-design.md`

### 6. AI evaluation (Evaluator)
- Input: scan image + task metadata (rubric, answer regions, target KCs)
- Output: structured `EvaluationResult` — per-problem quality tier (`mastered / shaky / needs-help / not-yet`), error pattern (if any), overall confidence
- Multimodal LLM (Claude Sonnet 4.6 primary; Gemini Flash fallback for cost)
- Scan → Debrief in ≤ 30 seconds (performance requirement)
- All evaluations are provisional until teacher-reviewed (Phase 1) or below confidence threshold (Phase 2+)

### 7. Debrief
- Digital-first: displayed on kiosk screen after submission; never triggers a print automatically
- Content: what went well, where things slipped (named KC), one or two specific suggestions, planned next task
- Tone: encouraging, age-appropriate, non-shaming, bilingual
- Printable on explicit student request; 0 Leaves
- Persisted to session record; visible to teacher and parent

### 8. Keyboard chat (Docent) — voice deferred to Phase 3
- Student can ask questions mid-task by keyboard
- Docent uses Socratic pattern: "what have you tried?" before giving hints — never gives direct answers
- Voice (Whisper STT + bilingual TTS; mic muted on idle): Phase 3
- Docent persona: friendly, encouraging, mildly playful, never sarcastic; bilingual

### 9. Teacher dashboard
- **Phase 1 (Observer):** Review queue — every AI evaluation awaiting approval; one-click approve/override + note
- **Phase 2 (Collaborator):** Flagged-only review; rubric refinement log; KC authoring surface
- Alert feed: students with ≥4 failed attempts on same KC
- Per-student radar chart view and session timeline
- Leaf grant button with reason dropdown
- Class Leaf summary and submission-rate metric
- Full spec: `/docs/pedagogy/teacher-direction.md`

### 10. BHCS portal integration
- Session reports pushed to parent portal inbox after teacher review
- Radar chart viewable from parent portal
- Leaf count in session report line: "earned 1 Leaf · 7 total this semester"
- Exhibit Gallery on student detail page (Phase 2.5+)
- No new auth surface — Atrium piggybacks on existing BHCS portal auth

### 11. Exhibits (creative contributions, Phase 2.5)
- Student can scan creative work (drawing, writing) without Leaf cost and without blocking the academic loop
- Exhibit stored with AI-extracted themes and interest signals
- Interest profile used as a soft hint in worksheet problem generation
- Gallery visible to parent (teacher controls `visible_to_parent` per Exhibit)

| Phase | Goal | Key gate |
|-------|------|----------|
| 0 — Scaffold | Flywheel runs end-to-end in dev with mock data | All services start; evaluator returns valid output |
| 1 — Vertical slice | One real student completes one real loop | Leaf gate live; BKT state persists |
| 2 — Teacher dashboard | Teacher can review all AI evaluations | Review queue + override live |
| 2.5 — Exhibits | Creative work scanned; interest profile influences task gen | Gallery in parent portal |
| 3 — Polish + voice | 6-week pilot ready | Scan → Debrief ≤ 30s; voice live |

---

## Out of scope (anti-goals)

- Content library / curated K–5 curriculum (LLM generates on demand)
- Gradebook, parent-teacher messaging, scheduling (live in BHCS portal)
- High-stakes assessment (AI grading is formative only)
- Printing without prior submission (never, regardless of payment or convenience)
- Open-ended homework-help chatbot
- Social Leaf transfer between students

---

## Success metrics (pilot, 6 weeks)

| Metric | Target |
|--------|--------|
| Submission rate (Cards submitted / Cards printed) | ≥ 0.85 |
| Scan → Debrief latency | ≤ 30s p95 |
| Teacher override rate | < 15% (signal of AI calibration quality) |
| Sessions per student per month | ≥ 4 |
| Mastery gain (mean Δ BKT probability across KCs) | Positive trend by week 4 |
